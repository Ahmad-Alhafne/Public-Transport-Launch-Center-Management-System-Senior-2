# AuditService API Contracts

Base path: /api/audit

Auth: Bearer JWT, role-based. All endpoints below require `Auditor` role except admin-only endpoints called out.

1) GET /api/audit/available-trips
 - Description: List upcoming trips available for auditing (unassigned or Available status).
 - Query: `?from=2026-06-07T00:00:00Z&to=2026-06-07T23:59:59Z&page=1&pageSize=50`
 - Response 200:
   {
     "trips": [
       { "tripId": "guid", "tripNumber": "T123", "route": "A→B", "departureTime": "2026-06-07T10:00:00Z", "vehicle": "Bus 12", "driver": { "id":"guid","name":"Driver" }, "remainingSeconds": 3600, "auditStatus": "Available" }
     ]
   }

2) POST /api/audit/pick-trip?tripId={tripId}
 - Description: Auditor requests to pick a trip. Requires `Auditor` role.
 - Behavior: DB-level atomic operation to create TripAudit with Status=Assigned if no active assignment exists. Returns Conflict if already assigned.
 - Responses:
   - 200 { "message": "Assigned", "assignedAt": "..." }
   - 409 { "message": "Trip already assigned" }

3) POST /api/audit/release-trip
 - Body: { "tripId": "guid" }
 - Description: Auditor releases assignment (sets TripAudit.Status=Cancelled or Available).

4) GET /api/audit/assigned-trip
 - Returns the current TripAudit for the calling auditor.

5) POST /api/audit/scan
 - Body (AuditRecordDto):
   {
     "bookingId": "guid",
     "citizenId": "guid",
     "tripId": "guid",
     "result": "Approved|Rejected|Duplicate|WrongTrip|InvalidQRCode",
     "notes": "optional"
   }
 - Behavior: Validate QR payload (BookingId+CitizenId+TripId+timestamp signature) via AuditService rules; store AuditRecord; enforce unique Approved per Booking+Trip.
 - Responses:
   - 200 { "message": "Recorded", "id": "guid" }
   - 409 { "message": "Already approved" }
   - 400 { "message": "Invalid QR" }

6) GET /api/audit/records?tripId={tripId}&page=1&pageSize=100
 - Returns paged list of AuditRecords for a trip.

7) GET /api/audit/history?auditorId={auditorId}&from=...&to=...
 - Admin-only: returns audits performed by auditor or global audit history.

8) GET /api/audit/stats?from=...&to=...
 - Admin-only: returns statistics: assignedTrips, completedAudits, totalScannedPassengers, todaysAudits etc.

Security & validation notes:
- JWT must include `sub` claim with user id and `role` claim. Audit endpoints validate `role: Auditor`.
- QR validation: prefer signed QR payload (HMAC over BookingId|CitizenId|TripId|Timestamp) generated by BookingService. AuditService should validate signature and timestamp (prevent replay by ensuring scan occurs before trip departure and/or by consulting BookingService for status).
- Prevent duplicate Approved scans by filtered unique index or application-level check inside a DB transaction.

Error model (common):
{
  "error": "BadRequest",
  "message": "Detailed message"
}
